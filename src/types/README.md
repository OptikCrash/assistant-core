# Types

This directory contains shared TypeScript type definitions used throughout the Assistant Core application.

## Overview

Centralizing type definitions provides:

- **Type consistency** - Same types used across modules
- **Single source of truth** - One place to update shared types
- **Better IDE support** - Autocomplete and type checking
- **Documentation** - Types serve as inline documentation
- **Refactoring safety** - Changes propagate through the codebase

## Files

### [messages.ts](messages.ts)

Types related to chat messages, task plans, and tool calls.

## Type Definitions

### ChatRequest

```typescript
interface ChatRequest {
  message: string;        // Natural language task description
  workspacePath?: string; // Optional path to workspace
  openFiles?: string[];   // Optional list of currently open files
}
```

The request payload sent to the `/chat` endpoint to initiate task planning.

**Fields:**

**`message`** (required)

- Natural language description of what the user wants to accomplish
- Examples:
  - "Add a migration to add an email column to users"
  - "Update the user model to include phone number"
  - "Generate a DTO for the posts endpoint"

**`workspacePath`** (optional)

- File system path to the workspace/project
- Used for context when executing tools
- Currently not utilized but available for future features

**`openFiles`** (optional)

- Array of file paths currently open in the editor
- Provides context about what the user is working on
- Can help the LLM make better decisions about related changes
- Currently not utilized but available for future features

**Example Usage:**

```typescript
const request: ChatRequest = {
  message: "Add validation to the user email field",
  workspacePath: "/Users/dev/my-project",
  openFiles: ["src/models/user.ts", "src/validators/user.ts"]
};

const response = await fetch('http://localhost:4000/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(request)
});
```

### ToolCall

```typescript
interface ToolCall {
  tool: string;                  // Name of the tool to execute
  input: Record<string, any>;    // Tool-specific input parameters
}
```

Represents a single tool invocation within a task plan.

**Fields:**

**`tool`** (required)

- Must match a tool name from the TOOL_REGISTRY
- Examples: `"read_schema"`, `"generate_migration"`, `"get_git_diff"`

**`input`** (required)

- Object containing parameters for the tool
- Structure varies by tool (validated by each tool's Zod schema)
- Must be a valid object (not null or primitive)

**Example Tool Calls:**

```typescript
// Read schema tool
const schemaCall: ToolCall = {
  tool: "read_schema",
  input: { table: "users" }
};

// Generate migration tool
const migrationCall: ToolCall = {
  tool: "generate_migration",
  input: {
    migrationName: "add_email_to_users",
    tableName: "users",
    columns: [
      { name: "email", type: "VARCHAR(255)", nullable: false }
    ]
  }
};

// Get git diff tool
const diffCall: ToolCall = {
  tool: "get_git_diff",
  input: { staged: true }
};
```

**Validation:**
The LLM is instructed to only use tools from the registry and to provide valid input objects. The OpenAI provider validates this before returning the plan.

### TaskPlan

```typescript
interface TaskPlan {
  intent: string;        // What the user wants to accomplish
  reasoning: string;     // Why this approach was chosen
  steps: string[];       // Human-readable execution steps
  toolCalls: ToolCall[]; // Actual executable tool invocations
}
```

The complete execution plan generated by the LLM.

**Fields:**

**`intent`** (required)

- High-level summary of the user's goal
- Written in clear, concise language
- Examples:
  - "Add email field to user model and database"
  - "Create migration for new posts table"
  - "Update DTO to include created_at timestamp"

**`reasoning`** (required)

- Explanation of why this approach was selected
- Helps users understand the AI's decision-making
- Examples:
  - "Adding a migration ensures database changes are tracked and reversible"
  - "Updating the Sequelize model maintains ORM consistency"

**`steps`** (required)

- Array of human-readable steps in execution order
- Helps users understand what will happen
- Used for displaying the plan to users before execution
- Examples:

  ```typescript
  [
    "Read current users table schema",
    "Generate migration to add email column",
    "Update Sequelize User model with email field",
    "Update User DTO for API responses"
  ]
  ```

**`toolCalls`** (required)

- Array of ToolCall objects to be executed
- Executed sequentially in order
- Each tool call is validated before execution
- Can be empty if no tools are needed (unusual)

**Example Complete Plan:**

```typescript
const plan: TaskPlan = {
  intent: "Add email column to users table",
  reasoning: "User needs to store and validate user email addresses for authentication",
  steps: [
    "Read the current users table schema",
    "Generate migration file to add email column",
    "Update User Sequelize model definition",
    "Update UserDTO for API consistency"
  ],
  toolCalls: [
    {
      tool: "read_schema",
      input: { table: "users" }
    },
    {
      tool: "generate_migration",
      input: {
        migrationName: "add_email_to_users",
        tableName: "users",
        columns: [{ name: "email", type: "VARCHAR(255)" }]
      }
    },
    {
      tool: "update_sequelize_model",
      input: {
        modelName: "User",
        fields: { email: { type: "STRING", allowNull: false } }
      }
    },
    {
      tool: "update_angular_dto",
      input: {
        dtoName: "UserDTO",
        fields: { email: "string" }
      }
    }
  ]
};
```

**Usage Flow:**

1. User sends ChatRequest to `/chat`
2. LLM generates TaskPlan
3. Plan is saved and returned with planId
4. User reviews the plan
5. User sends planId to `/execute`
6. System executes each ToolCall in sequence
7. Results are saved to audit log

## Validation

While these types provide TypeScript compile-time checking, consider adding runtime validation for API endpoints:

```typescript
import { z } from 'zod';

// Zod schema for ChatRequest
export const ChatRequestSchema = z.object({
  message: z.string().min(1, "Message cannot be empty"),
  workspacePath: z.string().optional(),
  openFiles: z.array(z.string()).optional()
});

// In route handler
chatRouter.post('/', async (req, res) => {
  try {
    const body = ChatRequestSchema.parse(req.body);
    // Now guaranteed to be valid
    const result = await createPlan(body);
    res.json(result);
  } catch (error) {
    if (error instanceof z.ZodError) {
      return res.status(400).json({ 
        error: "Invalid request", 
        details: error.errors 
      });
    }
    throw error;
  }
});
```

## Type Safety Best Practices

### 1. Avoid `any` Types

```typescript
// Bad
const data: any = req.body;

// Good
const data: ChatRequest = req.body;
```

### 2. Use Type Guards

```typescript
function isToolCall(obj: unknown): obj is ToolCall {
  return (
    typeof obj === 'object' &&
    obj !== null &&
    'tool' in obj &&
    'input' in obj &&
    typeof (obj as any).tool === 'string'
  );
}

// Usage
if (isToolCall(unknownData)) {
  // TypeScript knows it's a ToolCall here
  executeToolCall(unknownData.tool, unknownData.input);
}
```

### 3. Leverage Utility Types

```typescript
// Pick only certain fields
type ChatRequestPreview = Pick<ChatRequest, 'message'>;

// Make all fields required
type RequiredChatRequest = Required<ChatRequest>;

// Make all fields optional
type PartialPlan = Partial<TaskPlan>;

// Omit certain fields
type ToolCallWithoutInput = Omit<ToolCall, 'input'>;
```

### 4. Use Discriminated Unions

For types that can be different shapes:

```typescript
type PlanStatus = 
  | { status: 'pending'; planId: string }
  | { status: 'executing'; planId: string; progress: number }
  | { status: 'completed'; planId: string; executionId: string }
  | { status: 'failed'; planId: string; error: string };

function handlePlanStatus(status: PlanStatus) {
  switch (status.status) {
    case 'pending':
      // Only planId available
      break;
    case 'executing':
      // planId and progress available
      console.log(`Progress: ${status.progress}%`);
      break;
    case 'completed':
      // planId and executionId available
      break;
    case 'failed':
      // planId and error available
      console.error(status.error);
      break;
  }
}
```

## Testing with Types

Use types in tests for better type safety:

```typescript
import { TaskPlan, ToolCall } from '../types/messages';

describe('TaskPlan', () => {
  it('should create valid plan structure', () => {
    const toolCall: ToolCall = {
      tool: "read_schema",
      input: { table: "users" }
    };
    
    const plan: TaskPlan = {
      intent: "Read schema",
      reasoning: "Test",
      steps: ["Step 1"],
      toolCalls: [toolCall]
    };
    
    expect(plan.toolCalls).toHaveLength(1);
    expect(plan.toolCalls[0]).toMatchObject(toolCall);
  });
});
```

## Future Considerations

As the application grows, consider:

### 1. Separate Files

Split into multiple type files:

```text
types/
├── requests.ts    → ChatRequest, RefactorRequest, etc.
├── plans.ts       → TaskPlan, ToolCall, etc.
├── responses.ts   → API response types
└── index.ts       → Re-export all types
```

### 2. Type Generation

Generate types from OpenAPI schemas or database schemas:

```typescript
// Generated from OpenAPI spec
import { ChatRequest } from './generated/api-types';
```

### 3. Branded Types

Add type branding for IDs:

```typescript
type PlanId = string & { readonly __brand: 'PlanId' };
type ExecutionId = string & { readonly __brand: 'ExecutionId' };

// Prevents mixing up different ID types
function getPlan(id: PlanId) { /* ... */ }
```

### 4. Versioning

Add version fields for API evolution:

```typescript
export interface ChatRequestV2 extends ChatRequest {
  version: '2.0';
  streaming?: boolean;
}
```
